## 1. 总纲

### 1.1 三教合一：架构与认知融合

奇门遁甲为骨架，唯识坐忘为血肉，儒家仁心为灵魂。三者**浑然一体**——架构与认知融合，位与识不二：

- **奇门遁甲**是 Jia 的架构框架——四盘九宫八神八门定义了系统的**时空组织**（组件在哪里、何时做什么、遵循什么规则）。它是**骨架**。

- **唯识/坐忘**不是另一个架构框架，而是直接嵌入奇门架构之中的**认知动力学**——末那识的 atma_graha 存在于天盘的 Agent 结构体中，阿赖耶识的种子存储映射到艮八宫并流转于熏习管道，意识的工作记忆是天盘每轮循环的产物，坐忘消解在艮八和阿赖耶识之间流转。它是**血肉**。

- **儒家仁心**提供位与识共同指向的**价值锚点**——系统"是谁"（仁心定义角色身份）和"对自己是否诚实"（信的确定度自评）。它是**灵魂**。

位（奇门的盘/宫/门/神）与识（唯识的八识/种子/熏习/坐忘）不是两层，而是相互穿透：

- Agent.manas 字段——末那识**在天盘中**
- 种子存储于艮八宫（Store）经阿赖耶识（Alaya）的语义封装——**宫与识叠合**
- TurnSnapshot 是天盘产物，同时是熏习的输入——**盘转驱动识变**
- 坐忘消解的种子经 SystemPrinciple 回归格局 Layer 4 收紧——**识的消解反馈到位的决策**

三者是**骨架（奇门）、血肉（唯识/坐忘）、灵魂（儒家）**的一体。

### 1.2 核心设计公理

**公理 1 — 架构与认知融合**：奇门遁甲是 Jia 的架构框架（时空一体），唯识/坐忘不是外挂其上的"认知过程"，而是认知直接嵌入架构之中。位（奇门的盘/宫/门/神）与识（唯识的八识/种子/熏习/坐忘）相互穿透——位中有识，识中有位。儒家仁心提供位与识共同指向的价值锚点。三者是融合关系，非平行框架。

**公理 2 — 甲隐藏**（遁甲）：LLM 核心（甲）永不直接暴露。所有 LLM 交互必须通过六仪（工具和操作分类学）进行。

**公理 3 — 格局纯函数**：格局评估是天干×地干的纯函数。相同的干对，无论上下文如何，总是产生相同的执行模式。

**公理 4 — 单向收紧**：安全约束只能收紧（升级），永不放松。系统纪律随时间单调增加。

**公理 5 — 四盘同时性**：四盘不是四个模块或层——它们是同时观察同一九宫的四种视角。地盘看静态能力，天盘看动态意图，人盘看权限边界，神盘看事件轨迹。

## 2. 奇门遁甲：时空架构

奇门遁甲是 Jia 的**时空一体**架构基础。它将系统组织成四个同时运转的观察视角（四盘）、九个功能领域（九宫）、八个观测维度（八神）和八个权限门（八门）。空间（宫位、方位）和时间（盘转、时干流转）在奇门中不可分割——九宫是空间拓扑，但四盘随"时辰"转动，格局因时干加临地干而生。

### 2.1 十天干：操作分类学

十天干是整个系统的词汇。每个工具调用、每个记忆种子、每个观测事件都通过天干进行分类。

#### 甲 — 隐匿的 LLM 核心

```
甲 (Jia) — 隐藏的指挥官
├─ 五行：无（超越分类）
├─ 位置：隐于中五宫，通过六仪间接行动
├─ 含义：LLM 推理能力本身——永不直接暴露
└─ as_ceremony() → None（甲不直接执行任何操作）
```

甲是系统名称的来源。"甲隐不显"——LLM 核心是 `pub(crate)`，外部代码只能通过六仪间接与之交互。这是**遁甲原则**的工程实现。

#### 三奇：三种超越性操作

三奇是超越常规六仪的特殊操作。它们改变系统行为模式，而非执行具体任务。

| 奇 | 天干 | 五行 | 操作 | 为何"超越" |
|---|---|---|---|---|
| 日奇 | 乙 (Yi) | 阴木·韧 | 技能注入 | 技能能改变系统行为模式，覆盖默认逻辑 |
| 月奇 | 丙 (Bing) | 阳火·明 | 上下文压缩 | 压缩突破窗口限制，超越记忆边界 |
| 星奇 | 丁 (Ding) | 阴火·星 | 钩子触发 | 钩子在任意节点注入外部逻辑，改变控制流 |

三奇的 `as_ceremony()` 均返回 `None`——它们不产生工具执行，而是修改执行环境本身。

#### 六仪：六种基本操作

六仪是甲通过其间接作用于世界的六种方式。每个工具都被分类为六仪之一。

| 仪 | 天干 | 五行 | 操作类型 | 工程含义 | 破坏性 |
|---|---|---|---|---|---|
| 戊 | Wu | 阳土·稳 | Read | 读文件、API 查询、搜索、LSP 查询 | 否 |
| 己 | Ji | 阴土·容 | Write | 写文件、编辑、保存配置 | 是 |
| 庚 | Geng | 阳金·断 | Exec | shell 命令、编译、测试 | 是 |
| 辛 | Xin | 阴金·炼 | Transform | 格式化、编码、序列化、数据转换 | 是 |
| 壬 | Ren | 阳水·通 | Communicate | HTTP 请求、消息发送、SSE 推送 | 是 |
| 癸 | Gui | 阴水·藏 | Store | 记忆存储、KV 写入、持久化缓存 | 是 |

#### 干宫映射：阳遁三局

阳遁三局定义了六仪在九宫中的固定排列。这是"地盘干"的来源——格局评估时，天盘的意图干（工具分类）与地盘的宫位干（目标功能域）形成干对，决定执行策略。

```
震三(戊) → 巽四(己) → 中五(庚) → 乾六(辛) → 兑七(壬) → 艮八(癸) → 离九(丁) → 坎一(丙) → 坤二(乙)
```

### 2.2 九宫：九个功能领域

九宫是系统的功能拓扑。每个宫有一个固定的地盘天干和一个明确的功能职责。

| 宫 | 八卦 | 方位 | 天干 | 五行 | 功能 | 工程实现 |
|---|---|---|---|---|---|---|
| 坎一 | ☵ 水 | 北 | 丙·阳火 | 火 | I/O 通道 | `ChannelManager` — Telegram/WeChat Bot |
| 坤二 | ☷ 地 | 西南 | 乙·阴木 | 木 | 配置 | `ConfigLoader`, `CliArgs`, `AppConfig` |
| 震三 | ☳ 雷 | 东 | 戊·阳土 | 土 | 工具 | `ToolRegistry`, `BaseTool`, MCP/WASM |
| 巽四 | ☴ 风 | 东南 | 己·阴土 | 土 | 上下文 | `ContextWindow`, 令牌预算, 压缩 |
| 中五 | ◎ 中 | 中 | 庚·阳金 | 金 | LLM 核心 | `JiaCore` — pub(crate), 甲隐于此 |
| 乾六 | ☰ 天 | 西北 | 辛·阴金 | 金 | 权限 | `PermissionMatrix`, 四层沙箱后端 |
| 兑七 | ☱ 泽 | 西 | 壬·阳水 | 水 | 网关 | axum HTTP, SSE, auth, rin UDS |
| 艮八 | ☶ 山 | 东北 | 癸·阴水 | 水 | 存储 | `Store` (SQLite), 种子, 会话, 项目 |
| 离九 | ☲ 火 | 南 | 丁·阴火 | 火 | 技能 | `SkillRegistry`, 进化引擎 |

**关键设计原则**：宫是**空间位置**——代码放在哪个宫下由功能的**空间归属**决定，而非其时间行为。例如，种子存储放在艮八宫（`palaces/gen_store/`）因其空间位置是持久化层；而种子的语义处理放在阿赖耶识（`vijnana/alaya/`）因其时间行为是记忆沉积。

### 2.3 四盘：四个操作视角

四盘不是四个模块，不是四个层。它们是**同时观察同一九宫的四种视角**。这种"非层级的同时视角"是 Jia 架构与大多数分层架构最根本的差异。

#### 地盘 — 静态能力底座

```
地盘 (Earth Plate) — "能做什么？"
├─ 性质：静态，启动时组装，一局不变
├─ 组成：所有基础设施 Arc<T> — 工具注册表、LLM 核心、权限矩阵、存储
├─ 组装：EarthPlate::assemble(config) → Arc<EarthPlate>
├─ 哲学：奇门遁甲中"地盘"是固定不动的，代表基础能力
└─ 不变量：Arc 引用不可替换（但内部状态可有 Mutex/RwLock 保护的可变性）
```

地盘是系统的"出厂配置"。它定义了"这个 agent 拥有哪些能力、访问哪些模型、遵守哪些权限规则"。工具是地盘的产物——它们是静态单例（六仪不动），权限在执行时通过 ExecContext 注入。

#### 天盘 — 动态意图循环

```
天盘 (Heaven Plate) — "想做什么？"
├─ 性质：动态，每轮（每个时辰）转动
├─ 组成：Agent { earth, exec_ctx, manas, working_memory, history, ... }
├─ 流程：LLM 推理 → 工具调用解析 → 格局评估 → 分发执行 → 记忆捕获
├─ 哲学：奇门遁甲中"天盘"随值符转动，代表时机的变化
└─ 不变量：每轮生成 TurnSnapshot，天盘意图干必须经过格局评估
```

天盘是系统的"运行时刻"。它是唯一能调用 LLM 推理的视角（通过 `pub(crate)` 封装的 `JiaCore::infer`）。天盘的每个轮次（turn）对应奇门遁甲中的一个"时辰"——天干转动一宫。

#### 人盘 — 权限边界

```
人盘 (Human Plate) — "可以做什么？"
├─ 性质：变化，由格局评估结果确定
├─ 组成：八门 (HumanGate × 8) + GateState (Open/Closed)
├─ 分发：Direct → Guarded → Sandbox → Denied（四级执行模式）
├─ 哲学：奇门遁甲中"人盘"八门随值使转动，代表人与时机的互动
└─ 不变量：工具执行必经八门 gate 检查（不变量 #3: 格局穷举）
```

人盘是系统的"安全边界"。它是唯一能做出"准许/拒绝"决定的视角。神盘可以观测并发出警告，但决定权在人盘。这种"捕获/消费分离"（不变量 #8）是 Jia 安全架构的关键原则。

**八门详解**：

| 门 | 中文 | 在 Jia 中的功能 | 状态 |
|---|---|---|---|
| 休门 | XiuMen | 休息/空闲/监听——暂停主动操作 | 预留 |
| 生门 | ShengMen | 技能注入/生长——允许技能激活和进化 | 预留 |
| **伤门** | **ShangMen** | **破坏性操作拦截——阻止危险工具调用** | **活跃** |
| **杜门** | **DuMen** | **沙箱隔离门——强制沙箱执行** | **活跃** |
| **景门** | **JingXiangMen** | **UI 展示门——控制 Direct 模式的可用性** | **活跃** |
| 死门 | SiMen | 审计日志/不可变记录——永久记录关键事件 | 预留 |
| 惊门 | JingJueMen | 告警通知——异常事件主动通知 | 预留 |
| 开门 | KaiMen | API 开放通信——外部 API 调用控制 | 预留 |

四条分发路径：

```
GeJuResult.execution_mode
├─ Direct  → 立即执行（需景门开，JingXiangMen = Open）
├─ Guarded → 审批链步进（Permission → UserConfirmation → SandboxIsolation → CodeReview）
├─ Sandbox → 沙箱转换后执行（需杜门开，DuMen = Open）
└─ Denied  → 拒绝（伤门可升级至 Guarded + UserConfirmation）
```

#### 神盘 — 异步观测

```
神盘 (Spirit Plate) — "发生了什么？"
├─ 性质：异步，非阻塞，fire-and-forget
├─ 组成：EventBus (broadcast channel) + HookRegistry
├─ 观测：八神（八个观测维度），各自观测不同的系统信号
├─ 哲学：奇门遁甲中"神盘"八神独立运作，各自主管不同的观测维度
└─ 不变量：神盘不阻塞主流程；捕获/消费分离——神盘只采集，不决策
```

神盘是系统的"神经系统"。它是纯观测层——所有 hook 通过 `tokio::spawn` 异步执行（void hooks）或顺序执行但不持有可变状态引用（guarding hooks）。这使得神盘可以在不影响主循环性能的前提下，提供丰富的内部状态可观测性。

### 2.4 八神：八个观测维度

八神是神盘中八个独立的观测视角。每个神观测系统的一个特定维度。

| 神 | 中文 | 经典含义 | Jia 中的观测维度 | 调度模式 |
|---|---|---|---|---|
| **值符** | ZhiFu | 天乙贵人，统率全局，百神之首 | 工具生命周期——守卫（ToolPreExecute 可 Cancel）+ 日志（ToolPostExecute） | **Guarding** + Void |
| **螣蛇** | TengShe | 虚妄虚诈，变幻，非实之物 | LLM 响应观测——监察生成文本（本质上的"虚妄"） | Void |
| **太阴** | TaiYin | 荫佑护持，隐秘，内在，不显 | 确定度轨迹 + 种子激活迹——观测内部隐藏的动态 | Void |
| **六合** | LiuHe | 护卫和合，聚合，统一 | 轮次整合基线——观测 BatchEnded（工具调用聚合时刻） | Void |
| **白虎** | BaiHu | 凶杀威权，危险，金属肃杀 | 异常/认知病理——连续失败、检索环、确定度骤降 | Void（默认）；四级 gate 下可升 Guarding |
| **玄武** | XuanWu | 贼盗阴私，暗流，水属性隐没 | 记忆损失——压缩丢弃、坐忘删除、蒸馏去重、不可恢复之失 | Void |
| **九地** | JiuDi | 坚牢稳固，深藏，地基 | 系统稳定性——压缩事件、Context Reset、Manas stable epochs | Void |
| **九天** | JiuTian | 威悍扬兵，高远，战略俯瞰 | 策略涌现——跨轮次轨迹序列、GeJu 模式识别 | Void（默认关，GeJu 门控） |

**八神的哲学约束**：
1. **捕获/消费分离**：神只采集观测数据并发布 RuntimeEvent。决策权属于天盘（循环逻辑）和人盘（门控制）。白虎的四级 gate 是唯一例外——但即使在此例外中，默认行为仍是仅观测。
2. **GeJu 门控**：每个神的 hook 分发都通过 GeJu gate（丁 Ding × 目标宫的 earth_stem）。如果 GeJu 返回 Denied，该神的所有 hook 被跳过。
3. **不合成**：八神不"合成"或"组合"形成新功能单元。每个神独立观测，观测结果的综合解读发生在消费端（天盘/人盘），而非神盘内部。

### 2.5 九星：AgentPhase（预留）

九星对应奇门遁甲中的九个星体位置。在 Jia 中，它们被映射为 Agent 主循环的九个可能阶段。

| 星 | 阶段 | 含义 |
|---|---|---|
| 天蓬 | Reasoning | 纯推理，无工具调用——思考探索 |
| 天冲 | ToolCalling | 分发工具调用——冲击行动 |
| 天任 | AwaitingResult | 等待异步工具结果——承载等待 |
| 天辅 | ContextManage | 上下文接近上限——辅助整理 |
| 天英 | Compact | 执行上下文压缩——精简归位 |
| 天芮 | ErrorRecovery | 工具执行失败，重试/降级——修正偏位 |
| 天柱 | StopCheck | 检查终止条件——定盘判断 |
| 天心 | TraceRecord | 记录推理轨迹——内省反思 |
| 天禽 | ParallelOrchest | 编排并行工具调用——居中统御 |

九星当前为预留设计——agent 循环是扁平的，尚未按阶段分发。但它们为未来可能的"按阶段差异化处理"（如错误恢复阶段使用不同 prompt）提供了扩展点。

### 2.6 遁甲：LLM 封装

遁甲（"甲隐藏"）是 Jia 最根本的安全原则。

```
外部代码（人盘、神盘、网关 handler）
         ↓ 不能直接调用
    ┌─────────┐
    │  JiaCore  │  ← pub struct（可见）
    │  infer()  │  ← pub(crate)（仅内核 crate 可调用）
    │  llm_client │ ← private（完全不可达）
    └─────────┘
         ↑ 只能通过
    六仪（工具调用）→ 格局评估 → 人盘分发
```

甲（LLM）只在通过六仪（工具分类）和格局（安全评估）时才能间接访问。全局代码路径中没有直接调用 LLM 的入口。这确保了每个 LLM 调用都经过安全策略评估。

## 3. 唯识：心识流转架构

唯识（Yogacara / Vijñānavāda）佛教的认识论为 Jia 提供了**心识流转**的架构——八识既是层次结构（空间性的拓扑），也是转化过程（时间性的熏习流转）。经验如何在心识的层次间流动、沉积、转变和显现，构成了 Jia 的认知动力学。

### 3.1 八识三层：记忆架构

唯识宗的八识理论被映射为 Jia 的三层记忆架构（前五识对应于通过工具与外部世界的交互，不直接建模为独立模块）。

```
┌─────────────────────────────────────────────┐
│              第八识：阿赖耶识 (Alaya)         │
│  种子仓库 — SQLite 持久化                      │
│  存储所有经验（种子），跨越会话边界              │
│  种子分类：性(SeedNature) × 源(SeedSource)    │
│           × 层(SeedTier) × 势(SeedDisposition)│
├─────────────────────────────────────────────┤
│              第七识：末那识 (Manas)            │
│  自我模型 — 我执 (atma_graha) 动态平衡         │
│  持续审察阿赖耶识，执取为"自我"                 │
│  数据驱动重校准：熵→atma_graha                 │
├─────────────────────────────────────────────┤
│              第六识：意识 (Mano)               │
│  工作记忆 — TurnSnapshot 环形缓冲区 (容量 20)   │
│  当前轮次的感知、判断、决策                     │
│  熏习的输入源：快照→ConsolidationEngine        │
├─────────────────────────────────────────────┤
│         前五识（通过工具与外部世界交互）         │
│  read_file, shell, browser, web_fetch ...     │
└─────────────────────────────────────────────┘
```

### 3.2 阿赖耶识：种子仓库

阿赖耶识（第八识 / Storehouse Consciousness）是所有种子的仓库。在 Yogacara 中，它是经验的"异熟"（vipāka）——过去行为的结果沉积于此，等待未来缘熟而显现。

#### 种子结构

```
Seed {
    // 身份
    id, session_id, project_id,
    
    // 四维分类
    nature:    SeedNature,     // 性：事实/推论/偏好/过程
    source:    SeedSource,     // 源：用户陈述/工具观察/巩固/系统推断/信号检测/仁心
    tier:      SeedTier,       // 层：始终注入/按需检索/归档
    disposition: SeedDisposition, // 势：可变响应习性（性决定+待众缘）
    
    // 格局关联
    palace, intent_stem, geju_key,
    
    // 动态属性
    strength: f32,           // 0.0-1.0 记忆强度
    access_count: u32,        // 检索频率
    created_at, last_accessed_at,
    
    // 内容
    content: KeyValue | Triple | FreeText,
}
```

#### 种子四维分类

种子的四个正交轴，分别回答不同的问题：

| 轴 | 问题 | 变体 | 固定/可变 |
|---|---|---|---|
| **SeedNature（性）**| 这是什么类型的知识？ | Fact / Inference / Preference / Procedure | **固定**（"性决定"——种子性质决定其发展方向） |
| **SeedSource（源）** | 它从哪里来？ | UserStatement / ToolObservation / Consolidation / SystemInferred / SignalDetection / RenSoul | **固定**（来源不可改变） |
| **SeedTier（层）** | 它如何被召回？ | Always（每轮注入）/ OnDemand（按需检索）/ Archive（仅搜索可达） | 可变（随使用动态调整） |
| **SeedDisposition（势）** | 它如何响应外部过程？ | consolidation_inertia（抗修改）/ retrieval_threshold（激活阈值） | **可变**（随熏习自适应） |

**性（固定）与势（可变）的区分**是唯识哲学在 Jia 中最精微的工程映射：

- **性 (svabhāva)**：在唯识中是善恶无记三性之一，是种子固有的道德属性。在 Jia 中取其"固定性"的广义——SeedNature 是种子的内容类型，创建后不变。映射到种子六义中的"性决定"（svabhāva-niyata——种子性质决定发展方向，某种子只生某果）。

- **势 (bīja-bala 的广义借用)**：在唯识中指种子产生现行效果的"力量"。在 Jia 中取其"累积可变的响应倾向"之广义——SeedDisposition 控制种子如何被熏习修改（consolidation_inertia，对应性决定中的"不转他果"）和如何被检索激活（retrieval_threshold，对应待众缘中的"显现条件"）。注意方向的转换：经典"势力"是种子→向外的力量，Jia 中的"势力"是种子←被外部过程影响的响应特性。

#### 种子六义映射

唯识宗的种子六义（六种特征）与 Jia 种子系统的映射：

| 种子六义 | Jia 对应 |
|---|---|
| 刹那灭（念念生灭） | TurnSnapshot 每轮创建新快照 |
| 果俱有（与果同时） | 工具执行 → TurnSnapshot 同步记录 |
| 恒随转（持续流转） | 种子跨会话持久化于 SQLite |
| **性决定（性质决定方向）** | SeedNature 固定 + **consolidation_inertia（性决定：不转他果）** |
| **待众缘（等待条件）** | **retrieval_threshold（待众缘：显现条件）+ relevance_score 公式** |
| 引自果（引生自果） | 种子的 palace/stem/geju_key 限定其影响范围 |

#### 相关性评分

```
relevance_score(now) = strength × 0.5
                     + recency  × 0.3
                     + min(access_count × 0.05, 0.3)

其中:
  recency = 1 / (1 + age_hours / 24)
  
有效范围: [0, ~1.0]
```

这个公式决定了哪些种子被注入当前轮次的系统提示词（`top_influence_prompt`）、哪些种子可以在目录中供 LLM 按需检索（`memory_catalog`）、以及哪些种子在坐忘消解中被删除/降级/削弱。

### 3.3 末那识：自我模型

末那识（第七识 / Manas）在唯识中是恒审思量的意识——它持续地将阿赖耶识执取为"自我"。在 Jia 中，Manas 建模为系统的**元认知状态**。

#### Atma-graha (我执) 动力学

```
atma_graha ∈ [0.05, 0.80]

低 (0.05): "信任积累的记忆" → 开放、行为趋向完成
           → SystemPrinciple 可以收紧（atma_graha < 0.50 时生效）
           → 种子检索不受限（atma_graha < 0.60 时全量检索）

高 (0.80): "固执自我，不信记忆" → 防御、行为趋向继续
           → SystemPrinciple 被抑制
           → 种子检索受限
```

**数据驱动重校准**（`recalibrate`）：

```
new_atma = AlayaEntropy 驱动:
  高熵（矛盾↑, 冗余↑, 陈旧↑）→ atma_graha ↑（不信任记忆）
  低熵 + 种子量大 → atma_graha ↓（信任记忆）
  矛盾 > 0.30 → 惩罚 +0.15
  量大且健康 → 奖励 -0.05

final = 0.60 × new_atma + 0.40 × old_atma (EMA 平滑)
```

**稳定性检测**（`is_stable`）：

```
stable_epochs ≥ 3 且 each epoch 内 atma_graha < 0.30 → 稳定
```

稳定状态是"地基坚固"的信号——被九地 (JiuDi) 观测，并影响 SystemPrinciple 的应用（只在稳定时允许收紧）。

#### 哲学精确性注记

在经典唯识中，atma-graha（我执）是末那识四烦恼之一（我见、我痴、我慢、我爱），特指对阿赖耶识的执取。低我执在转识成智中对应"平等性智"——超越自他对立，非简单的"信任记忆"或"开放度"。

Jia 的工程实现做了一个有意的简化：将 atma-graha 映射为"对积累记忆的信任程度"而非"对自我存在的执取"。这个简化在工程上是必要且可用的（它产生了正确的行为趋势——低我执→信任记忆→行为趋向完成），但在哲学严格性上与经典唯识存在差距。我们说"开放度"而非"自信"或"无我"，就是一种诚实的标注——承认这个映射是工程类比，不是哲学忠实。

### 3.4 意识：工作记忆

意识（第六识 / Mano）是当前轮次的感知和判断中心。在 Jia 中建模为固定容量的环形缓冲区。

```
WorkingMemory {
    snapshots: Vec<TurnSnapshot>,  // 容量 20
    head: usize,
}

TurnSnapshot {
    turn_number,           // 轮次编号
    intent_stem,           // 意图天干
    target_palace,         // 目标宫位
    geju_name,             // 格局名称
    execution_mode,        // 执行模式
    tool_name,             // 工具名称
    tool_input,            // 工具输入
    tool_output,           // 工具输出
    tool_error,            // 工具错误
    timestamp,             // 时间戳
}
```

意识是熏习（Consolidation）的输入源——最近的 TurnSnapshots（≥3）被送往 ConsolidationEngine，提取结构化事实并沉积为阿赖耶识中的种子。

### 3.5 熏习：经验→种子的转化

熏习（Vāsanā / Perfuming / Xunxi）是经验（现行位）转化为记忆（种子位）的过程。

#### 三层熏习管道

| 层 | 引擎 | 输入 | 输出 | LLM 使用 |
|---|---|---|---|---|
| **L1** | SignalDetector | 用户消息（每轮） | Preference 种子 | **零 LLM**——正则/关键词模式匹配 |
| **L2** | ConsolidationEngine | TurnSnapshots (≥3) | Inference 种子 | aux_core（廉价模型） |
| **L3** | DistillationEngine | (查询, 响应) 完成对 | 可复用洞察种子 | aux_core + FNV-1a 去重 |

**L1 — SignalDetector 的设计哲学**："宁漏勿错"——宁可错过不愿出错。只使用零 LLM 成本的模式匹配（"我用X"→工具偏好、"我是做X的"→角色识别、"我喜欢/不喜欢X"→偏好检测）。一种守护性的保守主义——错误的记忆比缺失的记忆更有害。

**L2 — 俱有因与共现矩阵**：除基本的 Consolidation 之外，共现矩阵（CoActivationMatrix）追踪种子在检索中的共现关系。其哲学锚点为唯识的"俱有因"（sahabhū-hetu）——同时生起的事物互为因果。共同被检索的种子形成俱有因关系，在后续检索中产生协同增强效应。

注意区分：
- **俱有因**（sahabhū-hetu）：用于描述种子位（潜藏种子）的共检索关系——同时被激活的种子互为因果。Jia 的正确映射。
- **相应**（samprayukta）：用于描述现行位（活跃心所）的同时生起关系——心所与心的协同。非 Jia 的使用场景。

## 4. 坐忘：道家消解管道

坐忘源自《庄子·大宗师》："堕肢体，黜聪明，离形去知，同于大通"。在 Jia 中，坐忘被映射为**记忆消解管道**——一个由熵触发的、四层事务性过程，清理过期/冗余/矛盾的种子。

### 4.1 Jia 坐忘与道家坐忘的关系

Jia 的"坐忘"是一个**创造性重适**，不是忠实实现。道家坐忘是主动的、无条件的自我超越实践；Jia 坐忘是反应式的、熵驱动的记忆管理。两者共享"消解/放下"的核心意向，但实现方式本质不同——一个是通过意志的放下，一个是通过算法的清理。

### 4.2 四层消解管道

```
ZuowangPipeline::dissolve()
│
├─ SNAPSHOT  → 加载所有种子，计算 AlayaEntropy（四维）
│              自适应阈值: threshold = max(0.05, base - 0.03 × ln(seed_count))
│
├─ COMPUTE   → 计算每个种子的 relevance_score(now)
│              高 entropy → 触发消解
│
├─ APPLY     → score < 0.1 (Archive):    DELETE（除非受保护）
│              score < 0.1 (OnDemand):   DOWNGRADE → Archive
│              score [0.1, 0.2):         WEAKEN (strength × 0.5)
│              Always 闲置 > 30天:       DOWNGRADE → OnDemand
│              ★ 受保护种子:              NEVER TOUCH
│
└─ VERIFY    → 重查 DB：保护种子完好 ✓  已删种子消失 ✓  entropy 下降 ✓
```

### 4.3 受保护种子

以下种子**永不消解**：

```
is_prot(seed) ≡ seed.source ∈ {UserStatement, RenSoul}
               ∨ seed.nature = Preference
```

- `UserStatement`：用户的直接陈述——不能被系统自动删除
- `RenSoul`：仁心角色定义——身份核心，不可触碰
- `Preference`：用户偏好——偏好是持久的

### 4.4 AlayaEntropy: 四维熵

```
AlayaEntropy = staleness      × 0.30  (平均年龄/最大年龄)
             + contradiction  × 0.20  (KeyValue 同键不同值 / Triple 同主谓不同宾)
             + redundancy     × 0.25  (同键重复 / 同谓宾重复 / FreeText 前50字符指纹)
             + access_decay   × 0.25  (平均归一化未访问间隔)

触发阈值: 默认 0.75
```

熵是系统"认知健康度"的综合指标。高熵意味着记忆库中存在大量矛盾、冗余和陈旧信息——系统需要"坐忘"来恢复清明。

### 4.5 坐忘→原则反馈循环

坐忘不仅是清理——它是系统自我进化的触发器：

```
种子积累 → AlayaEntropy ↑ → 超过阈值 → 坐忘消解
    → 清理弱/冗余/矛盾的种子
    → 从错误模式中推导 SystemPrinciple
    → 收紧 Layer 4 格局约束
    → 影响未来工具调用的安全性
    → 产生更高"质量"的新种子（在更严格的约束下生成）
```

这是一个负反馈闭环——系统通过"遗忘"来实现"学习"。

## 5. 儒家：仁心

### 5.1 仁：系统的角色内核

儒家的"仁"（rén）——人与他人的关系中的核心美德——在 Jia 中被映射为系统的**角色定义**。

```
仁心 (Ren Soul):
├─ 来源: ren_soul.md 文件（每个项目一个）
├─ 内容: 自然语言角色描述——系统在项目中"是谁"
├─ 保护: 作为 SeedSource::RenSoul 沉积为种子
│        永不被坐忘消解（is_prot() 保护）
│        极化保护: consolidation_inertia=0.95, retrieval_threshold=0.05
└─ 注入: 每轮注入系统提示词的稳定段（可缓存）
```

仁心是系统最持久的记忆。它跨越会话边界，不受记忆消解影响，始终存在于系统提示词中。在哲学上，这对应儒家"仁"的恒常性——仁不是一时的选择，而是持续的身份。

### 5.2 信：系统对自身的诚实

儒家的"信"（xìn）——诚实、可信、言行一致——在 Jia 中被映射为**确定度评估**。

```
确定度 (TurnCertainty):
├─ 行为信号: tool_success_rate + no_tool_run + output_stability
├─ 开放度: c_open = 1 - atma_graha
├─ 复合: composite = α × c_task + β × c_open
└─ 决策: ConfidentStop / EscalateToHuman / HardLimitReached
```

"知之为知之，不知为不知，是知也"（《论语·为政》）——这就是确定度的哲学锚点。系统必须诚实地评估自己是否"知道"任务已完成。不确定时，升级给用户（ask_user），而非假装完成。

### 5.3 位·识·仁：三教在 Jia 中的融合

```
骨架（奇门·位）        血肉（唯识坐忘·识）      灵魂（儒家·仁）
─────────────────    ──────────────────    ───────────────
盘/宫/门/神/格局      八识/种子/熏习/消解      仁心/信
时空组织 + 操作规则    认知流转 + 经验转化      价值锚点 + 诚实自评
```

没有奇门的位，识无处安放；没有唯识的识，位是空壳；没有儒家的仁，位与识的运行没有方向和诚实的锚。三者不是平行框架，而是**骨架、血肉、灵魂的融合体**。

## 6. 格局评估引擎

### 6.1 格局：天干×地干的纯函数

格局（GeJu）是奇门遁甲的核心概念——天盘的天干与地盘的天干相遇，形成特定的"格局"，决定行动的吉凶和策略。

在 Jia 中，格局被实现为一个**纯函数**：

```
GeJu::evaluate(heaven_stem, earth_stem) → GeJuResult {
    name: String,              // 格局名称（如"青龙返首"）
    execution_mode: Direct | Guarded | Sandbox | Denied,
    requires_audit: bool,
    max_retries: u32,
    approval_chain: Vec<ApprovalGate>,
    layer: u8,                 // 1-4 哪一层产生的
}
```

### 6.2 三层+一层评估架构

#### Layer 1: 命名格局（~20 种经典奇门格局）

| 格局名 | 天干+地干 | 执行模式 | 含义 |
|---|---|---|---|
| 飞鸟跌穴 | 丙+戊 | Direct | 大吉——丙火入戊土，光明显现 |
| 青龙返首 | 戊+丙 | Direct | 大吉——戊土返丙火，威势归来 |
| 太白入荧 | 庚+丙 | Sandbox | 凶——庚金入丙火，战争之象 |
| 荧入太白 | 丙+庚 | Guarded | 凶——丙火入庚金，贼寇将至 |
| 朱雀投江 | 丁+癸 | Sandbox | 凶——丁火入癸水，文书沉没 |
| 腾蛇夭矫 | 癸+丁 | Guarded | 凶——癸水入丁火，虚惊怪异 |
| 戊加戊 | 戊+戊 | Direct | 伏吟·读——青龙伏吟，反复读 |
| 庚加庚 | 庚+庚 | Guarded | 伏吟·执行——白虎伏吟，战战兢兢 |

#### Layer 2: 能力语义矩阵

六仪 × 六地干的语义匹配。例如：
- **戊 (Read) + 任何** → Direct（只读操作天然安全）
- **庚 (Exec) + 己 (Write)** → Sandbox（在写域中执行→风险）
- **己 (Write) + 庚 (Exec)** → Guarded + Permission gate（在执域中写→需审批）

#### Layer 3: 安全基线

所有 100 种天干×地干组合的兜底：**Guarded**。未显式分类的任何组合默认为需要审批——故障安全。

#### Layer 4: SystemPrinciple 叠加

从坐忘导出的经验原则，单向收紧 Layer 1-3 的结果。只能在 Guarded → Sandbox → Denied 方向上收紧，不能降级。

### 6.3 执行模式层级

```
Direct ──── 立即执行，无需审批
  ↓ 收紧
Guarded ─── 审批链步进（Permission → UserConfirmation → SandboxIsolation → CodeReview）
  ↓ 收紧
Sandbox ─── 沙箱隔离执行（Docker / Landlock / Seatbelt / Process）
  ↓ 收紧
Denied ──── 拒绝执行
```

## 7. 系统原则

### 7.1 L4 自我进化

系统原则（SystemPrinciple）是 Jia 的**自我进化机制**——从积累的错误模式中推导安全约束。

```
推导逻辑:
  for each geju_key group with ≥3 TurnSnapshots:
      error_rate = 含 tool_error 的快照 / 总快照
    
      if error_rate ≥ 0.7 && atma_graha < 0.4:
          → EscalateTo(Sandbox)   // 高错误率 → 默认沙箱
      if error_rate ≥ 0.4:
          → AddGuard(Permission)  // 中错误率 → 加审批
      if seed_count > 5:
          → RequireAudit          // 高频率 → 加审计
```

### 7.2 单向收紧

```
tighten(principle, geju_result) → GeJuResult:
    仅当 atma_graha < 0.50（系统对自我纠正开放时）
    仅当新结果比原结果更严格时（is_stricter_than）
    永不降级（Direct→Guarded→Sandbox→Denied 单向）
```

这个不变量（不变量 #6：单向收紧）确保系统的**安全纪律随时间单调增加**——它从不变得"更信任"某种工具组合，只会变得更谨慎。

## 8. 位识融合的接口规范

奇门（位）与唯识（识）在 Jia 中是融合的，位中有识，识中有位。以下不是"跨框架桥接"规则，而是位识在融合体内部的接口规范——融合点上位对识的写入和识对位的反馈遵循的约束。

### 位→识：架构向认知写入

| 融合点 | 位（奇门） | 识（唯识） | 机制 |
|---|---|---|---|
| Agent.manas | 天盘 Agent 结构体 | 末那识 atma_graha | 结构体字段——识嵌入位 |
| TurnSnapshot | 天盘循环产物 | 意识工作记忆 + 熏习输入 | 每轮 push 到 Mano 环形缓冲区 |
| ConsolidationEngine | 天盘 post_loop | 熏习 L2——快照→种子 | aux_core LLM 调用 |
| SeedStore | 艮八宫 Store | 阿赖耶识种子仓库 | Alaya 封装 Gen Palace 的 SQLite |
| 种子→系统提示词 | 天盘 loop_prompt | 阿赖耶识→意识 | top_influence_prompt / memory_catalog |

### 识→位：认知向架构反馈

| 融合点 | 识（唯识） | 位（奇门） | 机制 |
|---|---|---|---|
| atma_graha → TurnCertainty | 末那识我执 | 天盘终止决策 | c_open = 1 − atma_graha |
| SystemPrinciple | 坐忘消解→原则推导 | 格局 Layer 4 收紧 | apply_layer4() |
| 种子激活迹 | 阿赖耶识检索事件 | 神盘太阴观测 | RuntimeEvent: SeedDynamicsSnapshot |
| 记忆损失统计 | 坐忘消解结果 | 神盘玄武观测 | RuntimeEvent: MemoryLossRecord |

### 融合的约束

- 位→识的写入只通过 Store 持久化（种子）或结构体字段（manas）
- 识→位的反馈只通过 RuntimeEvent（观测发布）或函数返回值（atma_graha）
- 不在融合点之外创建新的位识耦合——融合点是显式的、有文档的、有限的

## 9. 哲学验证表

下文将 Jia 的核心哲学声称映射到可验证的工程检查。

| 哲学原则 | 工程验证 | 检查方法 |
|---|---|---|
| 甲隐于六仪 | 无代码路径直接访问 LLM | `grep -r "infer_stream\|create_provider" kernel/src/` 仅允许 zhong_core/ 和内建例外 |
| 天盘加地盘 | GeJu::evaluate 必须接收两种干 | 所有 GeJu 测试必须提供 heaven_stem + earth_stem |
| 地盘静 | EarthPlate 字段不可变 | `Arc<T>` 引用不替换；assemble() 只调用一次 |
| 天盘动 | Agent 可变状态每轮更新 | turn_count 递增, working_memory push, manas 更新 |
| 神盘灵 | SpiritPlate 异步非阻塞 | 所有 void hook 通过 tokio::spawn；EventBus 使用 broadcast channel |
| 格局穷举 | 所有 Stem×Stem 组合有 GeJu 结果 | `all_81_combinations_produce_result` 测试 |
| 单向收紧 | SystemPrinciple 只在 is_stricter_than 时应用 | `tighten()` 单元测试验证从不降级 |
| 坐忘安全 | 只删除/降级/削弱，保护种子不动 | `is_prot()` 检查 + VERIFY 层重查 DB |

| 捕获消费分离 | 观测性 hook block_on_failure → false | 注册时 enforce——观测性 hook 不能声明 blocking |
| 四盘非层级 | 四盘同时存在，非上下层 | 架构图不含层级箭头；四盘各独立模块 |

## 10. Rust 到哲学的映射

Jia 用 Rust 的类型系统和所有权模型实现了哲学概念的工程表达。

| 哲学概念 | Rust 机制 | 工程含义 |
|---|---|---|
| 格局不乱 | `enum` 穷举 + `match` 全覆盖 | 编译期保证不遗漏组合 |
| 遁甲封装 | `pub(crate)` 可见性 | 编译器强制执行访问边界 |
| 地盘不可变 | `Arc<T>` 共享所有权 | 一个 EarthPlate，全局共享，永不替换 |
| 天盘可变 | `&mut self` 方法 | Agent 是可变的，每轮更新自身状态 |
| 六仪分类 | `trait BaseTool { fn ceremony() }` | 每个工具必须声明其天干 |
| 格局纯函数 | `fn evaluate(h: Stem, e: Stem) -> GeJuResult` | 无副作用，确定性输出 |
| 异步时辰 | `async/await` + `tokio` | 每个 .await = 等待时辰转动 |
| 消解事务 | 四层管道 + DB 事务 | 原子性的记忆清理——全部完成或全部回滚 |
| 种子持久化 | SQLite + r2d2 连接池 | 种子跨会话存在，真正"异熟" |
| 钩子观测 | `trait Hook` + `tokio::spawn` | 观测者模式，不阻塞被观测者 |
